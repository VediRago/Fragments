# Project Foundation

## Purpose

This file keeps the broad foundation of **Fragments** in one place.

It is not meant to replace act files, character files, prose, or deeper reference notes.

Its purpose is to define the current project-level logic clearly enough that future edits do not create ambiguity.

## Core Direction

Fragments is a story about people, connection, loss, recognition, and the way meaning can be felt before it is fully understood.

The project should not be presented as an encyclopedia of lore.

The clearest route is:

> people -> experience -> emotion -> behavior -> structure -> system

The reader should meet people first.

Systems, cosmology, and hidden structure matter because they create pressure, consequence, and recognition. They should not replace the emotional experience.

## Story Structure Question

The main structure question is:

> Can the same story lead different readers toward different conclusions without changing its events?

Different readers may notice different details, trust different patterns, or recognize different connections at different times.

The story should allow delayed recognition without making earlier scenes feel random.

## Core Concepts

### Self

The Self is the deeper blueprint from which Fragments emerge.

### Fragment

A Fragment is a different outcome explored from the same Self.

Fragments exist because the system explores possible outcomes.

Lucreta, Sera, Kael, and Erit are Fragments.

Project note:

> Fragments are not simply unrelated people, but the story should not require the reader to understand this immediately.

### Condition

The Condition exists in the Self.

Because Fragments originate from the same Self, the Condition can affect all Fragments.

The Condition causes early death.

This should stay in deeper reference unless the story earns the reveal.

## Prose Progression

Current act and part files are the authoritative source for act-specific information.

When writing prose, do not import act-specific details from previous acts unless continuity requires it.

Previous prose references should be used for comparison of prose progression, not as the final source of story content.

The story begins with more conventional prose.

It contains more description and more detail.

These are gradually reduced throughout the story.

The reduction happens through acts and parts.

The progression should be gradual enough that the reader does not consciously notice it at first.

By the time Unity begins, the reader should have already experienced a gradual reduction similar to what Kael experiences during Unity.

The progression should become visible mostly in hindsight.

## Title Logic

### Noeme

Noeme is an anagram of:

> One me

This supports the deeper unity / self / fragment logic.

The title logic should not force explanation too early.

It can work as a delayed-recognition device, where the title becomes clearer after the reader has lived through the story's structure.

## Clarity Rule

A project-level concept should never exist only because it sounds interesting.

If a concept matters, the file should explain:

- what it affects
- why it matters emotionally
- how it changes reader perception
- where it belongs in the story structure
- whether it is public-facing or deeper reference

If the idea cannot be explained clearly yet, mark it as unfinished instead of letting ambiguity look like canon.
